# `js/app.js` 职责拆分设计

## 背景

当前 `js/app.js` 约 1645 行，集中了承载应用壳、路由、列表/详情/编辑页渲染、弹窗、扫码、二维码生成和初始化流程的逻辑。这个结构已经让单次修改需要同时理解多个领域，新增功能时也容易互相影响。

## 目标

- 将 `app.js` 拆成职责清晰、边界稳定的多个模块。
- 降低单文件复杂度，便于后续定位问题和扩展功能。
- 保持现有行为不变，先做结构优化，再考虑进一步细分。

## 非目标

- 不重写数据层 `db.js`。
- 不改变现有 UI 样式和交互语义。
- 不在本次工作里顺手做业务重构或功能增强。

## 目标模块

### 1. `js/core/app-shell.js`

负责应用骨架：

- 维护 `state`
- 导航栈、返回逻辑、tab 切换
- 主 `render()` 调度
- 顶部标题、按钮、tab bar 的基础切换

### 2. `js/views/items.js`

负责物品相关页面：

- 物品列表
- 物品详情
- 物品编辑
- 物品关联页入口的页面级渲染协调

### 3. `js/views/containers.js`

负责容器相关页面：

- 容器树
- 容器详情
- 容器编辑

### 4. `js/views/alerts.js`

负责提醒页：

- 过期
- 临期
- 低库存

### 5. `js/scanner.js`

负责所有扫码能力：

- 摄像头申请与释放
- ZXing 解码循环
- 条码/二维码通用扫描入口
- 物品/容器关联扫描

### 6. `js/ui.js`

负责可复用 UI 构件：

- 弹窗
- 删除确认
- 实体管理器
- QR modal
- form helper

### 7. `js/bootstrap.js`

负责启动流程：

- 加载分类/标签
- 注册 Service Worker
- 种子数据
- 绑定事件
- 首次渲染

## 依赖关系

- `bootstrap` 依赖 `db.js`、`core` 和各功能模块。
- `core` 依赖 `ui` 和 `views`，但不包含具体业务页面细节。
- `views` 依赖 `db.js`、`ui.js` 和必要的 `scanner` 入口。
- `scanner` 依赖 `db.js`、`ui.js` 和 ZXing 全局库。
- `ui` 尽量保持无状态，只通过参数和回调工作。

## 数据流

1. `bootstrap` 完成数据预加载和环境初始化。
2. `core` 根据 `state` 决定当前 screen/tab，并调用对应 view。
3. `views` 只负责把数据渲染成 DOM，并把用户操作回调交回 `core` 或 `scanner`。
4. `ui` 提供对话框和表单组件，供各页面复用。

## 错误处理

- 启动失败时继续沿用现有的错误提示区，不静默失败。
- 扫码失败、相机不可用、找不到对象等情况保持现有提示文案和返回路径。
- 模块拆分后，跨模块依赖必须显式导出/导入，避免继续依赖隐式全局状态。

## 迁移策略

1. 先把 `app.js` 拆成新模块，但保留 `index.html` 现有脚本加载顺序。
2. 再把 `index.html` 中 `app.js` 切换为 `type="module"` 的入口加载方式。
3. 保留一层轻量入口，确保初始化流程和页面行为不变。
4. 如后续仍然偏大，再继续把 `views/items.js` 内部拆成更小的子文件。

## 测试策略

- 手动回归主路径：物品列表、容器树、提醒页、详情页、编辑页、扫码、二维码展示。
- 检查首屏加载、返回按钮、tab 切换、删除确认、保存流程是否正常。
- 重点验证相机授权、相册扫码和 PWA 离线启动不被拆分破坏。

## 验收标准

- `app.js` 只保留薄入口或明显减少到较小体量。
- 业务逻辑按职责分散到多个文件，单文件职责清晰。
- 现有功能行为保持一致，没有新增回归。
